Arquitectura

Tres procesos, un motor propio en Rust y cero dependencias externas para el usuario final.

🗺️ Visión general

flux-browser.exe   ← proceso principal Rust
  ├── chrome_view    ← WebView2 para la UI React del browser
  ├── content_view   ← WebView2 para las páginas web
  └── flux-backend   ← sidecar Node.js · SQLite local · datos del usuario

flux-engine (Rust)

Motor principal. Pipeline HTML/CSS/JS, seguridad, BM25 ranking, servidor Axum. Sin Node.js, sin Electron.

UI React

Chrome del browser en React + TypeScript + Tailwind. Corre en chrome_view (WebView2). Comunicación via IPC bidireccional con wry.

flux-backend (Node.js)

Sidecar Express + Prisma + SQLite. Gestiona historial, favoritos, auth JWT, Gemini IA. Distribuido como exe standalone.

🤔 Por qué esta arquitectura

Los navegadores más exitosos no construyeron su propio motor de renderizado desde el día 1 — construyeron su valor por encima de uno existente:

BrowserValor real propioSuperficie de render
BraveAd blocking · Rewards · PrivacyChromium (Blink)
ArcSpaces · AI · UX innovadorChromium
Firefox FocusPrivacy core · Tracking protectionWebKit
FluxMotor Rust propio · Search · Privacy · BM25 · JS sandbox · CSPWebView2 / WebKit (provisional)

El motor existe y funciona hoy. Lo que falta es conectar el FluxSoftRenderer al content_view de la ventana nativa — ese es el siguiente paso del roadmap. Cuando ese cable esté hecho, Flux será uno de los pocos browsers con motor de píxeles escrito en Rust.

⚙️ flux-engine — Motor Rust

El núcleo de Flux. Escrito completamente en Rust. No usa Electron, no usa Chromium, no usa código C++.

Pipeline HTML/CSS/JS

URL
 └→ Fetcher (reqwest + rustls)
     ├→ SecurityLayer.check_url()
     │     · HTTPS upgrade (HSTS preload)
     │     · Ad/tracker block
     │     · CSP del servidor
     └→ HTML
          └→ Tokenizer (zero-copy, &str sin allocar)
               └→ Parser (DOM arena — Vec<Node> + NodeId)
                   ├→ CSS Cascade (especificidad, herencia, UA stylesheet)
                   ├→ JS Runtime (QuickJS sandbox)
                   │     · DOM bindings completos
                   │     · fetch() real HTTPS + CSP connect-src
                   │     · 16 MB heap · 512 KB stack
                   └→ Layout Engine
                        └→ Display List
                             └→ FluxSoftRenderer
                                   Pixel buffer XRGB8888
                                   Glyph atlas (fontdue)
                                   softbuffer (DXGI/Metal)

Decisiones de memoria

TécnicaBeneficio
&'a str en tokens y DOM0 copias del HTML fuente en memoria
Vec<Node> arena + NodeId(usize)Sin Box/Rc, cache-friendly, sin fragmentación de heap
SmallVec<[Attr; 4]>Atributos en stack si son ≤ 4 — sin heap allocation
ComputedStyle planoSin heap, cache-line friendly
LayoutBox Vec contiguoSin overhead por nodo, iteración eficiente

🔌 Referencia IPC

Comunicación bidireccional entre la UI React y el motor Rust usando wry IPC. La UI envía comandos, el motor responde con eventos.

Comando IPCDirecciónDescripción
navigateUI → RustNavegar a una URL en el content_view
reloadUI → RustRecargar la página actual
go_back / go_forwardUI → RustHistorial de navegación del WebView
zoom_in / zoom_out / zoom_resetUI → RustControl de zoom de la página
minimize / maximize / closeUI → RustControl de la ventana nativa
chrome_heightUI → RustAjuste dinámico de la altura del chrome
mute_tabUI → RustSilenciar / activar audio de una pestaña
download_mediaUI → RustIniciar descarga con yt-dlp
download_progressRust → UIPorcentaje, velocidad y bytes de la descarga activa
permission_requestRust → UISolicitud de permiso (cámara, mic, geolocalización)
permission_responseUI → RustRespuesta del usuario al diálogo de permiso
url_changedRust → UILa URL del content_view cambió (actualiza la barra)
page_titleRust → UITítulo de la página cargada

🦀 Seguridad por diseño — Rust vs C++

El 70% de las vulnerabilidades de Chrome y Firefox son errores de memoria en C++. Rust los elimina en compile time.

VulnerabilidadC/C++ (Chrome/Firefox)Rust (Flux)
Buffer overflow Posible Imposible en safe Rust
Use-after-free Posible Imposible — borrow checker
Null pointer Posible Imposible — no hay null
Data races Posible Imposible en compile time

🔨 Compilar desde el código fuente

Requisitos: Rust 1.75+, Node.js 18+, Windows 10/11 con WebView2

# 1. Clonar
git clone https://github.com/tu-usuario/flux-browser.git
# 2. Descargar yt-dlp (una sola vez)
.\scripts\download-yt-dlp.ps1
# 3. Compilar el backend
.\scripts\build-backend.ps1
# 4. Compilar el browser
cd flux-engine
cargo build --release --bin flux-browser
# 5. Ejecutar el motor solo (demo píxeles)
cargo run --bin flux-render

Variables de entorno del backend (flux-backend/.env)

DATABASE_URL=file:./flux.db            # SQLite local
JWT_SECRET=cambia_esto_por_un_secret_largo
GEMINI_API_KEY=tu_api_key_aqui           # para IA, OCR, canciones
PORT=[configurable]
